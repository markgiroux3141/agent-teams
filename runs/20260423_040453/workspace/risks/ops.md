# Ops failure modes — checkout refactor pre-mortem

## Failure modes

- **v2 dashboard monitoring gap** (likelihood: high, impact: high)
  Mechanism: Checkout dashboards in production track v1 success rate, p99 latency, error rate per endpoint. v2 dashboards are still draft. The new coupon endpoint `/v2/coupons/apply` has no dedicated panels for success rate, latency percentiles, error breakdown, or cache-hit rate. If the coupon service has a subtle performance degradation on Monday ramp (e.g., p99 latency 200ms → 2s, or 2% error rate on specific discount types), the dashboard won't surface it — the metric gets aggregated into generic checkout stats and is invisible. On-call will not see what's degrading.
  Early signal: None visible in Datadog. Support-ticket volume spike in Slack/Zendesk is the first signal (the silent degradation pattern we've seen twice before, 20–60 min after impact starts).

- **Runbook for v2 does not exist** (likelihood: high, impact: high)
  Mechanism: Current checkout runbook documents v1 endpoints, v1 CLI commands, v1 dashboards, and v1 failure recovery steps. When a v2 incident happens Friday night or over the weekend (thin coverage), the on-call engineer is reading instructions for the wrong system. They will query the wrong Datadog dashboard, run the wrong curl commands to test v1 endpoints, escalate to the wrong backend oncall, and waste 15–30 minutes in the wrong directions before realizing the runbook is stale. This delay compounds when the system is under stress and the engineer is already context-switching between v1 and v2 support requests.
  Early signal: On-call engineer asks for help in Slack; escalation requests come slow or to wrong people. The gap doesn't trigger any pager or metric.

- **Silent failure on Redis session store failover** (likelihood: medium, impact: high)
  Mechanism: Session Redis is single-node per region with async replication and tested failover (last drill two months ago). If Redis fails over on Sunday night or Monday morning while the flag is ramping, the async replication means there is a window where writes are not guaranteed to the standby. During that window, session objects are lost or return stale state. v2 checkout is session-heavy (coupon choice, payment method selection are stored in Redis). Users see "session expired" or "invalid payment method" errors. Because there is no alert on "session store unavailable" errors for v2, the issue stays silent for 20–60 minutes. By the time support tickets spike and we page on-call, the error rate is already high enough that manual flag flip back to v1 takes longer than desired.
  Early signal: 403 / "session not found" error spike in checkout error logs. Only visible if we have an alert on v2 session errors (we don't yet). Otherwise: support tickets.

- **Feature flag ramp stalls or returns stale state** (likelihood: medium, impact: high)
  Mechanism: The feature flag control plane is used to ramp from 0% (Friday ship) to 100% (Monday). If the flag service becomes unavailable, is slow to propagate state changes, or returns inconsistent values across regions, the ramp will not work as intended. Scenario 1: flag service is down or returning errors. Scenario 2: some regions are on v2 (ramped) while others are still on v1 (stale data in control plane), causing inconsistent user experience. Scenario 3: on-call tries to flip the flag OFF as an emergency rollback, but the control plane is slow or partitioned, and some customers continue to use v2 for 10+ minutes. None of these scenarios have a monitoring alert set up. On-call will not know the flag state is in question until users complain.
  Early signal: Flag evaluator latency or error rate spike (visible in feature-flag control-plane metrics). Or: users on the same account see different checkout flows (one v1, one v2), signaling inconsistent flag state.

- **Thin on-call coverage on Friday night into weekend ramp** (likelihood: high, impact: medium)
  Mechanism: Friday ship is 16:00 UTC. I (ops) am primary on-call for the weekend. Backend engineer is secondary. The team is thin — no other ops, no dedicated incident commander, no secondary ops pager. If a silent degradation starts at 18:00 Friday (2 hours after ship), and the first signal is a support-ticket spike at 18:45 (support notices, escalates to Slack), I am the only ops responder. If I am unavailable (in another incident, asleep), there is a 20–60 minute window with zero ops visibility into what is happening. The secondary (backend engineer) is not trained on ops dashboards and may not escalate correctly. Escalation and decision lag directly increases the time to restore or rollback.
  Early signal: Support-ticket volume spike in Zendesk (20–60 min after impact). Slack mentions from support team. If there is no alert, ops does not know there is an issue until a human tells them.

## Mitigations I'd invest in this sprint

- **v2 dashboard monitoring gap**: Add v2-specific dashboard panels to Datadog THIS WEEK before Friday ship. Panels required: (1) coupon endpoint `/v2/coupons/apply` success rate and error count, (2) coupon endpoint p50/p95/p99 latency, (3) error breakdown by type (timeout, 4xx, 5xx), (4) Redis session cache-hit rate. Link the dashboard in the updated v2 runbook.

- **Runbook for v2 does not exist**: Update the checkout on-call runbook THIS WEEK to include v2-specific sections: (1) v2 endpoint paths and meanings (especially `/v2/coupons/apply`), (2) v2 error codes and what they signal, (3) v2 dependencies and which services to page (coupon service owner, payments backend, session Redis), (4) v2 dashboard link and which panels to check first, (5) manual test steps using v2 API (curl examples), (6) v2 rollback procedure (flip feature flag OFF and verify old v1 path works).

- **Silent failure on Redis session store failover**: Set up a PagerDuty alert on v2 session-store errors (403 / "session not found" rate > 0.5% for 2 minutes). Add a separate alert on Redis replication lag > 5 seconds. Run a 15-minute Redis failover drill Friday afternoon after ship to verify alerts fire and on-call knows the recovery steps (manual failover promotion, flag flip to v1 if needed).

- **Feature flag ramp stalls or returns stale state**: (1) Add metric alert on feature-flag control-plane latency > 500ms or error rate > 1% (page ops-checkout). (2) Document flag control-plane failover steps in runbook (which team owns it, how to force a rollback, how to query flag state manually). (3) Schedule a 10-minute flag failover test Friday at 17:00 UTC (after ship, before ramp starts) to verify the alert triggers and on-call can recover.

- **Thin on-call coverage on Friday night into weekend ramp**: (1) Brief the secondary (backend engineer) on what a "silent degradation" looks like operationally and how to escalate to ops (Slack mention #ops-checkout, page ops-primary if no response). (2) Set up a bridge Slack channel (#checkout-launch-ops) for the weekend where support can escalate directly; ops reads it every 15 minutes as a backstop if formal paging fails. (3) Schedule a check-in call Friday 18:30 UTC (30 min after ship) with ops primary + engineer + support lead to verify dashboards are live and everyone knows the signal path.
